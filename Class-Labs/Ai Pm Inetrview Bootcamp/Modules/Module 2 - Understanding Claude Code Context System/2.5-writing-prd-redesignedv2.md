# Lesson 2.5: Writing Dual PRD (Human + Agent) Using Context [v2]


---


## Overview


In previous lessons, you completed market research and user research. Now you'll write **dual PRDs** — one for humans (stakeholders) and one for agents (automation systems) — following a structured four-phase process.


**The Four Phases:**
| Phase | What You Do |
|-------|-------------|
| **Phase 1: Thinking** | Decompose the problem into Goals, Tasks, Agent Steps, and an E2E scenario *before writing anything* |
| **Phase 2: Drafting** | Generate Human PRD (Markdown) + Agent PRD (JSON), each with functional and non-functional requirements |
| **Phase 3: Refinement** | Use Claude Code skills to chain context, edit sections, enforce templates, and document the tool manifest |
| **Phase 4: Evaluation** | Audit for missing requirements, weak assumptions, edge cases, agent failure points, and handoff contracts |


**Why start with Thinking (Phase 1)?**


Writing the E2E scenario *first* grounds every Goal, Task, and Agent Step in a real user flow before decomposition begins. Without this anchor, PRDs accumulate isolated story fragments that don't add up to a working product.


**What You'll Build:**
- `outputs/thinking-lease-compliance.md` — Problem decomposition + annotated scenario
- `outputs/prd-lease-compliance.md` — Human-readable PRD (~3000–4000 words)
- `outputs/prd-lease-compliance-agent.json` — Agent-consumable PRD (structured JSON)
- `outputs/task-graph-visualization.md` — Visual task graph


**What You'll Learn (8 Claude Code Skills):**
Context chaining, cross-referencing, iterative refinement, template customization, prompt engineering, output validation, version management, and context priority resolution.

## PHASE 1: THINKING


> **Why this phase matters:** Writing the E2E scenario first grounds Goals, Tasks, and Agent Steps in a real user flow before any decomposition begins. Example: Jennifer, 150 leases, audit in 10 days → upload to auditor delivery.


### Step 2: Decompose the Problem


Before writing any PRD, prompt Claude to structure your thinking:


**Prompt:**
```
Before writing the PRD, help me decompose the problem for the Lease Compliance Reporting feature.


Read @outputs/market-research.md and @outputs/user-research.md first.


Then produce a structured thinking document with four sections:


1. GOALS — What does success look like?
  - What user problem are we solving? (be specific: who, what pain, what frequency)
  - What business outcome are we targeting? (metric, magnitude, timeline)
  - What does failure look like?


2. TASKS — What must the product do?
  - List the core product capabilities required (not features — capabilities)
  - For each: is this user-facing, system-facing, or both?


3. AGENT STEPS — For each task, decide: ML vs. rule-based?
  - Which steps require ML/AI reasoning?
  - Which steps can be handled with deterministic rules?
  - Which steps involve agent-to-agent handoffs?
  - What are the failure risks at each handoff?


4. END-TO-END SCENARIO — Write this FIRST before any decomposition:
  One annotated scenario: Jennifer (Legal Team Lead), 150 leases, audit deadline in 10 days.
  Walk through every step from file upload to auditor delivery.
  For each step: what does Jennifer do? What does the system do? Which component handles it?
  Flag edge cases inline (e.g. what if a lease file is malformed?).


Save to: outputs/thinking-lease-compliance.md
```


**💡 What's Happening Here?**


This is **Phase 1: Thinking** — the step that was missing from the original lab. You are:
- **Grounding your goals** in measurable outcomes before writing requirements
- **Writing the E2E scenario first** so every downstream requirement maps to a real user moment
- **Deciding ML vs. rule-based** per step before committing to a technical approach in the PRD
- **Surfacing handoff risks** before they become engineering surprises


**Expected Output:** `outputs/thinking-lease-compliance.md` — a structured decomposition (~500–800 words) that you will reference throughout the rest of this lab.

## PHASE 4: EVALUATION (Human PRD)


### Step 7: Evaluate Human PRD for Quality and Completeness


> **Why this matters:** Most PRD reviews only check for missing sections. This evaluation checks for *thinking quality* — weak assumptions, unhandled edge cases, and requirements that sound complete but will fail in production.


**Prompt:**
```
Evaluate outputs/prd-lease-compliance.md for quality. This is NOT just a template check — go deeper.


Run four evaluation passes:


PASS 1 — Missing Requirements:
- Compare against @templates/prd-template-redesigned-v2.md — list any missing sections
- Are all user personas from user research represented in user stories?
- Are all market opportunities from market research addressed?
- Are NFRs fully defined with specific numbers (not vague statements)?
- Is the E2E journey complete from first user action to final outcome?


PASS 2 — Weak Assumptions:
- List every assumption in the Risks & Assumptions section
- For each: is it stated clearly? Is it testable? Is there evidence supporting it?
- Flag any assumption that is unverified and high-stakes (e.g. "users will adopt self-serve reporting")
- Identify implicit assumptions not listed but embedded in requirements


PASS 3 — Edge Cases:
- Review the E2E user journey — what unhandled inputs exist?
- What happens if a lease file is malformed, partially complete, or in an unsupported format?
- What happens if an accounting system integration is unavailable mid-report?
- What if a compliance standard changes after report generation?
- Are any of these edge cases addressed in requirements? If not, they need user stories or NFR entries.


PASS 4 — Requirement Clarity:
- Flag any requirement that is vague or unmeasurable (e.g. "fast performance", "secure data handling")
- Suggest specific, testable rewrites for each vague requirement


Provide a structured evaluation report. Then fix all identified issues directly in outputs/prd-lease-compliance.md.
```


**💡 What's Happening Here?**


This is **Phase 4: Evaluation** — upgraded from a simple template check to a rigorous quality audit:
- **Pass 1** catches structural gaps
- **Pass 2** catches reasoning gaps (unvalidated assumptions that will cause rework)
- **Pass 3** catches product gaps (unhandled inputs that will cause user escalations)
- **Pass 4** catches language gaps (vague requirements that engineers will interpret inconsistently)


**Expected Output:** Evaluation report + updated PRD with all identified issues resolved.

## PART B: Generate Agent-Consumable PRD


### Step 8: Generate Agent PRD (JSON)


Now create the machine-readable version for automation systems:


**Prompt:**
```
Generate an agent-consumable PRD in JSON format based on the human PRD (outputs/prd-lease-compliance.md).


Follow the schema in @templates/prd-schema-v2.json exactly.


Include ALL sections:


1. metadata: title, status, author, date, version, human_prd_reference (set to "outputs/prd-lease-compliance.md")


2. problem: statement, insights[] with source file:line_number, impact (structured object)


3. goals: success_metrics[] with baseline/target/timeline, guardrail_metrics[], leading_indicators[], lagging_indicators[]


4. scope: in_scope[], out_of_scope[] with item + reason


5. user_stories[]: All stories with persona, story text, acceptance_criteria[], priority (P0/P1/P2), complexity


6. e2e_journey: Extract the Jennifer scenario — scenario_name, context, steps[] (step, user_action, system_response, component, notes), edge_cases[]


7. nfrs: ALL categories as typed objects:
  - performance: {api_response_time_ms_p95, report_generation_time_s, target_volume, max_concurrent_users}
  - security: {encryption_at_rest, encryption_in_transit, authentication, access_control, compliance_standards[]}
  - data_retention: {retention_period_years, pii_handling, deletion_policy}
  - availability: {uptime_sla_percent, rto_hours, rpo_hours, backup_frequency}
  - accessibility: {wcag_level}


8. risks: assumptions[], known_risks[] with likelihood/impact/mitigation


9. agent_workflow:
  a) task_graph — nodes[]:
     - lease_data_extraction (DataIngestionAgent, no dependencies)
     - data_validation (ValidationAgent, depends_on: [lease_data_extraction])
     - compliance_check (ComplianceAgent, depends_on: [data_validation], can_run_parallel_with: [risk_scoring])
     - risk_scoring (RiskScoringAgent, depends_on: [data_validation], can_run_parallel_with: [compliance_check])
     - report_generation (ReportAgent, depends_on: [compliance_check, risk_scoring])
     - audit_trail_logging (AuditTrailAgent, depends_on: [report_generation])


  b) handoff_contracts[] — For EVERY agent boundary (5 contracts total):
     Each contract must include: from_agent, to_agent, input_schema (typed fields), output_schema (typed fields), invariants[], failure_modes[]


  c) tool_manifest[] — For EVERY tool used:
     Each entry must include: tool_id, tool_name, owning_agent, step, type, idempotent (true/false), timeout_ms, retry_strategy, fallback


10. go_no_go_criteria[]: Boolean checklist for ship decision
   Include at minimum:
   - All P0 user stories have passing acceptance tests
   - NFR performance targets verified in load test
   - Security audit completed (SOC 2 controls validated)
   - All handoff contracts validated end-to-end
   - Legal review of IFRS/GAAP compliance logic complete
   - Beta user feedback addressed


11. milestones: phases[] with dates, rollout object


Format: Pretty-printed JSON with 2-space indentation


Save to: outputs/prd-lease-compliance-agent.json
```


**💡 What's Happening Here?**


Key concepts you are encoding:
- **Task Graph (DAG):** Compliance Check and Risk Scoring can run in parallel — reducing total pipeline execution time
- **Handoff Contracts:** Typed schemas prevent silent failures where Agent A passes malformed data and Agent B proceeds incorrectly
- **Tool Manifest:** Idempotency flag determines retry behavior — a non-idempotent tool retried will cause duplicate actions
- **go_no_go_criteria:** Ship checklist that every agent can evaluate without interpreting prose


**Expected Output:** Complete agent PRD JSON (~600–900 lines).

## PHASE 4: EVALUATION (Agent PRD)


### Step 10: Evaluate Agent PRD for Correctness and Safety


> **Why this matters:** Agentic pipelines most commonly fail at handoffs. Agent A completes successfully, passes a malformed object, Agent B guesses and proceeds incorrectly — silently. This evaluation catches those failure points before they reach production.


**Prompt:**
```
Evaluate outputs/prd-lease-compliance-agent.json for correctness and safety.


Run four evaluation passes:


PASS 1 — Schema Compliance:
- Does it follow @templates/prd-schema-v2.json structure?
- Are all required fields present and correctly typed?
- Is human_prd_reference set correctly?
- Are all NFR sub-objects fully populated (no empty strings or null values)?


PASS 2 — Task Graph Validation:
- Check for circular dependencies — the graph must be a valid DAG (no cycles)
- Are all task IDs referenced in depends_on valid (do they exist as nodes)?
- Are parallel execution opportunities correctly identified?
- Is there any task that could start earlier but is blocked unnecessarily?


PASS 3 — Handoff Contract Audit:
- Is there a handoff contract for EVERY agent boundary in the task graph?
- For each contract:
 - Is input_schema complete? List any missing required fields.
 - Is output_schema complete?
 - Are invariants specific and testable (not vague)?
 - Are failure_modes defined for both hard failures (malformed input) and soft failures (partial data)?
- Flag any agent boundary with no contract — this is a silent failure risk.


PASS 4 — Tool Manifest Audit:
- Is every tool used in the task graph listed in the manifest?
- For each non-idempotent tool: is retry_strategy set to "none"? If not, flag as a duplicate-action risk.
- Are timeouts set for all external API calls?
- Does every tool have a fallback defined?


PASS 5 — go_no_go Completeness:
- Does the go_no_go_criteria list cover: P0 features, performance validation, security, handoff contracts, and legal/compliance?
- Are there any criteria where must_pass is false but should be true for a compliance product?


Provide a structured evaluation report. Then fix all issues directly in outputs/prd-lease-compliance-agent.json.
```


**💡 What's Happening Here?**


Phase 4 for the Agent PRD focuses on **structural correctness and failure prevention**:
- **Pass 1** ensures the JSON is valid for downstream agent consumption
- **Pass 2** ensures the task graph has no deadlocks or missed parallelism
- **Pass 3** is the most critical — handoff contracts are where most agentic pipelines fail silently
- **Pass 4** prevents duplicate actions from unsafe retries on non-idempotent tools
- **Pass 5** ensures the ship checklist reflects the compliance risk profile of this product
